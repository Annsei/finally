# Phase 4 — UI Design Contract

> Visual and interaction contract for the Frontend Portfolio & Trading phase (FE-09–FE-15).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Aesthetic target: dark, data-dense Bloomberg-style trading terminal. Desktop-first.
> **Inherits the entire Phase 3 contract** (`03-UI-SPEC.md`). This document extends it with the
> new panels: main ticker chart, portfolio heatmap, P&L chart, positions table, trade bar, and AI chat.

| Property | Value |

## Spacing Scale

8-point scale, inherited unchanged from Phase 3. Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Cell inner padding, chat badge inner padding, gap between trade inputs and buttons |
| sm | 8px | Positions-table row vertical padding (compact), heatmap tile inner padding, chat message bubble vertical gap |
| md | 16px | Panel inner padding (chart panel, chat panel, trade bar container) |
| lg | 24px | Gap between stacked center-column panels (chart → heatmap/P&L row → trade bar → table) |
| xl | 32px | Reserved (major layout breaks) |
| 2xl | 48px | Reserved |
| 3xl | 64px | Reserved |

**Layout column widths (CONTEXT D-01, fixed pixel values — not spacing tokens):**
- Watchlist column: ~200px fixed
- Center content column: `flex-1`
- Chat column: ~320px fixed (collapses to a thin toggle strip when collapsed, CONTEXT D-09)
- Column gap: `gap-4` (16px) — matches existing `index.tsx` layout

**Exceptions:**
- Positions-table rows are compact (~28–32px height) to match the Phase 3 Bloomberg-style watchlist density: `8px` vertical padding on `14px` / line-height-1.2 data text (CONTEXT Discretion → follows Phase 3 D-08).
- Heatmap tile minimum width: `64px` so small positions stay clickable even when their weight-derived width% is tiny (CONTEXT specifics — "minimum tile width").
- Trade bar buttons: minimum `44px` touch target height for Buy / Sell tap targets.

## Typography

Inherited from Phase 3 — three roles, two weights. No new sizes or weights introduced. Chat prose is the one place line-height relaxes.

| Role | Size | Weight | Line Height | Family | Usage |
|------|------|--------|-------------|--------|-------|
| Display | 20px | 600 (semibold) | 1.2 | JetBrains Mono | Inherited — header portfolio total value only. Not reused for new panels |
| Body / Data | 14px | 400 (regular) | 1.2 | JetBrains Mono | Positions table cells (ticker, qty, avg cost, current price, P&L, %), heatmap tile values, trade input values, P&L/main chart axis numbers |
| Label | 12px | 600 (semibold) | 1.2 | system sans | Column headers (positions table), panel captions, trade input field labels, chat badge text, heatmap empty-state |
| Chat prose | 14px | 400 (regular) | 1.5 | system sans | Chat message bubble text only — the sole long-form prose in the app, so it takes the 1.5 body line-height for readability |

**Weights:** exactly two — 400 (regular) and 600 (semibold). No others (matches Phase 3).

**Line height:** 1.2 for all tabular/terminal data (unchanged). 1.5 is permitted **only** inside chat message bubbles, which contain multi-line AI prose. This is the documented density exception for Phase 4.

**Tabular figures:** `font-variant-numeric: tabular-nums` (Tailwind `tabular-nums`) on every numeric column in the positions table, heatmap tiles, trade bar, and chart axes — so values do not shift horizontally as digits change on live update.

## Color

60/30/10 split, inherited from Phase 3. Every value below already exists in `tailwind.config.js` under `terminal.*`. Do not substitute or add hexes.

| Role | Token | Value | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `terminal-bg` | `#0d1117` | App background, all panel base surfaces (chart, table, trade bar, chat) |
| Secondary (30%) | `terminal-surface` | `#1a1a2e` | Panel insets, chat message bubble background, selected/hover row lift, heatmap container backdrop |
| Accent (10%) | `terminal-accent` | `#ecad0a` | See reserved-for list below — strictly limited |
| Blue | `terminal-blue` | `#209dd7` | Main ticker chart line, sparkline line (inherited), neutral chart strokes, P&L chart line (neutral baseline) |
| Purple | `terminal-purple` | `#753991` | Submit/primary-action buttons: chat send button, and (optionally) a neutral default state — see trade button note below |
| Up / profit | `terminal-up` | `#22c55e` (green) | Buy button, positive P&L text, profit heatmap tiles, uptick flash, P&L chart area when total value ≥ baseline |
| Down / loss | `terminal-down` | `#ef4444` (red) | Sell button, negative P&L text, loss heatmap tiles, downtick flash, P&L chart area when total value < baseline, chat sell-action badge |
| Amber | `terminal-amber` | `#f59e0b` | Reconnecting status (inherited). Not used in Phase 4 panels |
| Text primary | `terminal-text` | `#e6edf3` | Default text on dark surfaces (no pure white) |
| Text muted | `terminal-muted` | `#8b949e` | Column headers, secondary labels, flat/zero values, placeholder text in trade/chat inputs |
| Border | `terminal-border` | `#30363d` | Row dividers, panel borders, input borders, heatmap tile gridlines |

**Accent (`#ecad0a`) reserved for — explicit list:**
- The 2px left accent bar on the currently selected watchlist row (inherited, CONTEXT D-10 / Phase 3).
- The currently selected ticker's emphasis in the main chart header/title (the chart for the selected ticker), echoing the selection signal.
- The buy-action badge border in the chat panel (CONTEXT specifics: "accent yellow border for buy actions").

Accent yellow is **never** used for general buttons, body text, table values, hover states, chart lines, or status dots. It remains the selection/emphasis signal plus the chat buy-badge accent.

**Trade button colors (FE-13, semantic):**
- **Buy** button → green `#22c55e` (profit/long semantic). Sole green button in the app.
- **Sell** button → red `#ef4444` (loss/short semantic). This is the destructive-direction action.
- Purple `#753991` is the submit color for the **chat send** button (non-directional submit). Per PLAN.md §2 purple is the "submit button" color; trade buy/sell use semantic green/red because direction carries meaning, chat send is neutral submit → purple.

**Heatmap color intensity (CONTEXT D-07, specifics):** Map P&L% magnitude to opacity over the green/red base:
- Profit tiles: `rgba(34, 197, 94, α)` where α scales from `0.3` (small gain) to `1.0` (large gain).
- Loss tiles: `rgba(239, 68, 68, α)` where α scales from `0.3` (small loss) to `1.0` (large loss).
- Flat / zero P&L tile: `terminal-surface` `#1a1a2e` with muted text.
- Tile text stays `terminal-text` `#e6edf3` and must remain legible at α = 1.0 (use a subtle text shadow or keep semibold).

**Chat action badge colors (CONTEXT D-10, specifics):**
- Buy action → accent yellow `#ecad0a` border on `terminal-surface` fill.
- Sell action → red `#ef4444` border on `terminal-surface` fill.
- Watchlist add/remove → neutral `terminal-muted` `#8b949e` border on `terminal-surface` fill.

**Semantic up/down:** Green/red remain the only semantic pair beyond accent — carrying profit/loss, uptick/downtick, and buy/sell direction everywhere (flash, change%, P&L text, heatmap, trade buttons, P&L chart).

## Interaction Contract

| Interaction | Trigger | Visual Contract |
|-------------|---------|-----------------|
| Ticker select → main chart | Click watchlist row (Phase 3 `onSelectTicker`) | Main chart re-renders for the new ticker; chart title shows ticker in accent `#ecad0a`. On load with no selection, first watchlist ticker auto-selected (CONTEXT D-03) |
| Ticker select → trade bar autofill | Same click event | Trade bar ticker input syncs to `selectedTicker` via `useEffect` (CONTEXT D-12); user may still type a different ticker |
| Positions current-price flash | SSE tick for a held ticker (`useTicker`) | Reuse Phase 3 `.animate-flash-up` / `.animate-flash-down` on the current-price cell, 500ms fade. Flat = no flash |
| P&L value color | On portfolio recompute | Green text if unrealized P&L ≥ 0, red if < 0, muted `#8b949e` if exactly 0. Applies per-row and to any total |
| Heatmap tile render | Portfolio data load / refresh | Tile width from weight%; fill green/red at P&L-derived α; flat = surface. Hover: subtle border lift to `terminal-blue`, cursor pointer |
| Buy submit | Click Buy button | Optimistic SWR `mutate('/api/portfolio')` with computed state, then `POST /api/portfolio/trade` `{side:"buy"}`, then revalidate (CONTEXT D-13). Button disabled + shows pending state during request |
| Sell submit | Click Sell button | Same optimistic flow with `{side:"sell"}`. This is the destructive-direction action — see copywriting |
| Trade error | API returns validation error (e.g. insufficient cash) | Inline red `#ef4444` message below inputs, `12px` label type; cleared on next submit attempt (CONTEXT D-14). No modal, no toast |
| Chat send | Submit chat input (Enter or Send button) | Purple `#753991` send button; on submit show loading indicator (animated dots or spinner) until full `POST /api/chat` response arrives — no token streaming (CONTEXT, ROADMAP) |
| Chat action confirmation | Assistant response contains `trades` / `watchlist_changes` | Render badge/pill below the assistant bubble: buy=accent border, sell=red border, watchlist=muted border (CONTEXT D-10). Visually distinct from the prose bubble |
| Chat history load | Component mount | Fetch `GET /api/chat` (last 20 msgs, asc by `created_at`); render existing conversation. Empty → show empty state |
| Chat panel collapse | Click collapse toggle | Panel slides/fades to a thin toggle strip; toggle re-expands. Open by default (CONTEXT D-09). Collapse animation: Claude's discretion (slide preferred) |
| P&L poll | Every 30s timer | Re-fetch `GET /api/portfolio/history`, append/refresh the P&L chart without layout shift (CONTEXT D-16) |

## Copywriting Contract

Terse, factual, terminal tone. No exclamation marks. Inherits Phase 3 empty/error feed copy for the watchlist.

| Element | Copy |
|---------|------|
| Primary CTA (trade) | `Buy` and `Sell` — single-verb buttons in the trade bar (FE-13) |
| Secondary CTA (chat) | `Send` — chat message submit button (FE-14) |
| Trade input labels | `Ticker` · `Qty` (12px sans labels above/beside inputs) |
| Trade input placeholders | Ticker: `AAPL` (or current selection) · Qty: `0` |
| Positions table headers | `Ticker` · `Qty` · `Avg Cost` · `Price` · `P&L` · `Change %` |
| Positions empty state | Heading: `No positions yet` · Body: `Use the trade bar to buy shares.` (CONTEXT D-08) |
| Heatmap empty state | Reuses positions empty copy: `No positions yet. Use the trade bar to buy shares.` (CONTEXT D-08) |
| P&L chart empty state | `No portfolio history yet.` — shown before the first snapshot is available |
| Chat input placeholder | `Ask FinAlly about your portfolio…` |
| Chat empty state | `Ask FinAlly to analyze your portfolio, suggest trades, or manage your watchlist.` |
| Chat loading indicator | Animated dots or `Thinking…` label while awaiting `POST /api/chat` |
| Chat buy badge | `Bought {qty} {ticker} @ {price}` |
| Chat sell badge | `Sold {qty} {ticker} @ {price}` |
| Chat watchlist badge | `Added {ticker}` / `Removed {ticker}` |
| Trade error (insufficient cash) | `Insufficient cash for this order.` (inline, below inputs) |
| Trade error (insufficient shares) | `You don't own enough shares to sell.` (inline) |
| Trade error (invalid input) | `Enter a valid ticker and quantity.` (inline) |
| Destructive confirmation | **None.** Sell is a destructive-direction action but, per PLAN.md §2, trades execute instantly with **no confirmation dialog** (deliberate: simulated money, fluid demo). Sell button colored red `#ef4444` as the only destructive signal; errors surface inline. AI-executed trades also auto-execute without confirmation (PLAN.md §9) |

**Tone:** Terminal-terse for data and errors; the chat panel is the one conversational surface and may use natural sentences in the AI's own responses (the AI controls that text, not this contract).
